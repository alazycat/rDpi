Issue #4 — Phase 8.2: 新增 MongoDB 协议检测 — 2026-07-15

Design Decisions

Decision BSON 顺序扫描而非构建完整解析树

Ambiguity: MongoDB 使用 BSON 作为 wire 协议序列化格式，有多种嵌套结构。

Choice: 实现一个有限状态 BSON 扫描器，按类型-名字-值的顺序线性扫描，遇到 isMaster/hello 字段即退出。

Rationale: 当前只关心握手阶段的几个字段（isMaster、version、maxWireVersion、maxMsgSizeBytes），不需要处理嵌套文档或数组。线性扫描 + 未知类型安全跳过比完整 BSON 解析器简单得多，同时显著降低假阳性。
参见 nDPI 的 ndpi_search_mongodb 实现，它同样采用有限 BSON 扫描方案。

Decision isMaster 字段出现即为检测成功，不要求 ok 字段

Ambiguity: SPEC 同时参考 isMaster 和 ok 两个字段。但 OP_QUERY 请求方向只有 `{isMaster: 1}`，没有 ok 字段。

Choice: isMaster/hello 字段值非零即可认为检测成功。ok 字段不再作为必要条件（但仍会被正确解析）。

Rationale: MongoDB isMaster 是一个标准命令，在任何 MongoDB 连接中都会出现。字段名足够特异，不太可能随机出现在非 MongoDB 的 BSON 中。双向（请求和响应）检测提高了检出率。

Deviations

None — implementation fully follows the SPEC and PRD as written.

Tradeoffs

Tradeoff 复用 `database` feature 而非新建独立 feature

Alternatives: (A) 新建 `nosql` feature；(B) 复用现有的 `database` gate。

Chosen: B。MongoDB 与 MySQL/PostgreSQL/Redis 同属数据库协议组，复用 `database` 无需用户额外启用。如果未来 MongoDB 的 BSON 解析复杂度增加，可以再拆分。

Tradeoff 未知 BSON 类型安全跳过表

Alternatives: (A) 未知类型直接 break；(B) 未知类型尝试根据固定大小跳过。

Chosen: B。对于 ObjectId(12)、Timestamp(8)、Decimal128(16) 等固定大小的类型，在跳过表中列出。对于无法确定大小的类型（如 Symbol、code_w_s），直接 break 中止扫描。

Rationale: 在有足够字段用于判断后，提前中止不会降低检测率。跳过表平衡了覆盖率和安全性。

Open Questions

Question OP_REPLY 格式在现代 MongoDB 版本中是否仍然使用？

MongoDB 3.6+ 默认使用 OP_MSG 替代 OP_QUERY/OP_REPLY。OP_REPLY (opCode=1) 处理保留为旧版本兼容。

建议: 无需进一步处理。如果后续发现 OP_REPLY 流量已经绝迹，可以移除该分支。
