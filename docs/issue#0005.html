Issue #5 — Phase 8.3: 新增 WireGuard 协议检测 — 2026-07-15

Design Decisions

Decision 最小长度校验使用保守（偏大）的值

Ambiguity: WireGuard 协议没有固定包长，各种消息类型的实际最小长度取决于加密认证标签的大小。

Choice: Initiation ≥148, Response ≥128, CookieReply ≥64, Transport ≥32 字节。

Rationale: 选取保守值确保只匹配完整的 WireGuard 消息，降低在随机 UDP 数据上的假阳性。如果后续发现假阴性率过高，可以适当降低。

Decision `vpn` feature 与 `iot` 对称设计

Ambiguity: WireGuard 使用独立 feature gate 还是合并到现有 gate？

Choice: 新建 `vpn` feature（Cargo.toml + types.rs 中 cfg-gated）。

Rationale: 为 OpenVPN、IPsec 等未来 VPN 协议预留统合 gate。与 `iot`（MQTT 预留）对称设计。

Deviations

None — implementation fully follows the SPEC and PRD as written.

Tradeoffs

Tradeoff 仅检测握手消息而非传输数据

Alternatives: (A) 仅检测 Handshake Initiation (type=1)； (B) 检测所有四种消息类型。

Chosen: B。Transport Data (type=4) 最小长度只有 32 字节，虽然假阳性风险略高，但加上 reserved 全零检查 + 端口辅助后，风险可接受。覆盖所有消息类型能显著提高检出率。

Open Questions

None — implementation is straightforward and no unresolved questions.
