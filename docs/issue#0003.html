Issue #3 — Phase 8.1: 新增 MQTT 协议检测 — 2026-07-15

Design Decisions

Decision MQTT v5 Properties 使用 VBI 跳过而非解析

Ambiguity: MQTT v5 在可变头尾部引入了 Properties（Variable Byte Integer + 键值对列表）。SPEC 未指定是否解析 Properties 内容。

Choice: 解码 Property Length（VBI）后整体跳过，不逐个解析 Properties 条目。

Rationale: 当前需求仅做协议识别和基础元数据提取（ClientID, Will Topic），不依赖 Properties 内容。跳过策略降低了代码复杂度，避免了处理数十种 Property 类型的开销。后续如果需要提取 Session Expiry Interval 等字段，可增量添加。

Decision 端口辅助仅影响置信度，非硬条件

Ambiguity: SPEC 提到 MQTT 使用标准端口（1883/8883），但未说明非标准端口是否拒绝检测。

Choice: 标准端口 → Dpi 置信度；非标准端口 → DpiPartial 置信度。两种情况下均尝试 DPI 检测。

Rationale: 遵循 rDpi 现有模式（如 WireGuard 检测器设计），非标准端口的 MQTT 部署（如企业内部自定义端口）仍需可检测，仅是置信度降低。

Deviations

None — implementation fully follows the SPEC and PRD as written.

Tradeoffs

Tradeoff 预检查 (payload[0] & 0xF0) 在 detector 和 parser 中重复

Alternatives: (A) 仅在 parser 中检查； (B) 在 detector 中预检查 + parser 中再检查。

Pros/Cons: A 更 DRY，B 性能更优——快速拒绝非 CONNECT 包（如 HTTP GET），避免进入耗时的解析。

Chosen: B。预检查是简单的单字节操作，开销可忽略，但能在最外层快速拒绝 15/16 的非 MQTT 流量。

Tradeoff Will Topic 提取后不跳过 Will Message

Alternatives: (A) 读取 Will Message 长度后前进 offset； (B) 不读取，解析器返回后 offset 不再使用。

Chosen: B。因为当前 parser 函数在 Will Topic 后不再使用 offset，不读取 Will Message 不影响正确性。但需要注意：如果未来 parser 扩展为读取更多字段（如 Username），Need 重新处理 offset 的连续性。

Open Questions

Question MQTT 非 CONNECT 消息（PUBLISH, SUBSCRIBE）是否应增加检测支持？

当前实现仅检测 CONNECT 消息。如果 MQTT 流的第一包不是 CONNECT（如中间截获的流量），检测会失败。是否需要增加对 PUBLISH/SUBSCRIBE 等固定头类型的识别作为补充？

建议: 当前范围（L2 CONNECT 检测）符合 Phase 8 的 Out of Scope 界定。如需提高检测率，可在后续迭代中增加 PUBLISH 消息识别。
